# 检索模块未来规划
本文档是[ROADMAP](../ROADMAP.md)中的任务2的ROADMAP

## 整体架构

```
Query
  ↓
BM25 双路召回（定位索引 + 内容索引，各 top K）
  ↓
合并去重（≤ 2K 条候选）
  ↓
Dense Retrieval 召回（预留）
  ↓
RRF 融合（BM25 + Dense 多路结果合并排序）
  ↓
Reranker 精排
  ↓
最终 top K 结果
```

## 1. BM25 双路召回（已完成）

在 Retrieval 阶段，对定位索引和内容索引分别召回 top K 条结果，合并去重后送入下游。

- **定位索引**：`"文件名 章节 条款号"` 格式的文本，分词后可用于按法规结构定位
- **内容索引**：条款正文文本，用于按语义内容检索

两路各召回 K 条（而非 K/2），充分召回后合并去重，避免因截断丢失候选。最终由 Reranker 统一排序，无需在 Retrieval 阶段做加权融合或意图判断。

## 2. Dense Retrieval + RRF 融合（已完成）

引入向量检索，与 BM25 形成互补：
- BM25 擅长精确关键词匹配
- Dense 擅长语义相似度匹配

使用 RRF（Reciprocal Rank Fusion）将 BM25 和 Dense 的排序结果融合，取长补短。

## 3. 中期：Reranker 精排（已完成）

### step3.1（已完成）
索引持久化：BM25（bm25s save/load）+ Dense（faiss write/read）+ metadatas（JSON），启动时检测已有索引直接加载。索引存放在 `artifacts/index/` 下，按组件分为 `bm25/` 和 `dense/` 子目录。
### step3.2（已完成）
在融合结果之上加一层精排模型（Qwen3-Reranker-0.6B），对候选集做二次打分，输出最终排序。

Reranker 的排序能力替代了原本在 Retrieval 阶段做分数加权的需求——让专业的层做专业的事。

## 4. 后期：Profile 优化

### 4.0 重构优化（已完成）

**Dense 检索 GPU 编码**：Dense 检索的 query 编码已默认使用 GPU（`DENSE_DEVICE="cuda"`），耗时从 ~150-260ms 降至 ~18ms（~10x 加速）。
编码设备通过 config.py 的 `DENSE_DEVICE` 可配置，可选 `"cpu"` 但速度会显著下降（~15x）。

**索引构建与检索分离**：索引构建逻辑从 `__init__` 迁移到 `build()` 类方法，检索只通过 `load()` 加载磁盘索引。
新增独立的 `index.py` 构建入口，`main.py` / `server.py` 不再包含构建逻辑。
删除了 `_loaded_from_disk` 标志，设备选择由 config 显式控制。

**Reranker batch_size 可配置**：`RERANKER_BATCH_SIZE` 提升到 config.py，调用方通过参数传递，不再硬编码在方法签名中。

### 4.1 知识库长度统计与切分策略

#### 背景

当前知识库中存在极端长文档（最长 14851 字符），带来两个问题：

1. **Rerank OOM 风险**：Qwen3-Reranker 为 LLM 架构，`padding=True` 导致 batch 内所有序列 pad 到最长文档长度，显存与 `batch × seq_len²` 成正比。长文档导致 batch_size 无法调大（batch=15 即 OOM），拖慢整体精排速度。
2. **下游生成约束**：Rerank 输出的文档将送入推理模型做生成，总字符数需控制在 ~8K 以内（输出 4 条文档，每条 ~2048 字符）。超过此长度的文档即使召回也没有实际价值，反而增加推理时延并可能干扰主模型效果。

#### 4.1.1 知识库 content 长度统计（已完成）

统计 `artifacts/index/metadatas.json` 中所有 content 字段的长度分布，输出：
- 基础统计：max、min、mean、median、P90、P95、P99
- 分段统计：各长度区间的文档数量和占比
- 极端值清单：超过阈值（如 1000、2000、4000 字符）的文档数及其内容摘要

#### 4.1.2 智能切分策略（已完成）

对超过阈值的文档进行递归切分，在索引构建时实施。切分后需重建索引。

**适用范围：** 仅对超过 `CHUNK_SIZE`（默认 512 字符）的文档执行切分。根据统计，受影响文档约 117 条（3.8%），对索引规模影响很小。

**三管线设计：制表符表格 / 密集清单 / 普通文本**

长文档实际分三类，各有不同的语义结构，需走不同的切分管线：

| | 制表符表格 | 密集清单 | 普通文本 |
|---|---|---|---|
| **检测方式** | content 含 `\t` | 无 `\t`，首行含 `序号` | 其他（默认） |
| **文档数** | ~9 条 | ~7 条 | ~101 条 |
| **切分方式** | 按 `\n` 按行分组 | 按 `\n` 按行分组 | 优先级分隔符递归切分 |
| **Overlap** | 无 | 无 | 仅低语义分隔符时启用 |

**典型示例：**

1. **制表符表格** — GA801 检验表（14,851 字符，127 行）：
   ```
   序号\t项目\t合格要求
   1\t车辆识别代号\t汽车、摩托车...应具有唯一的车辆识别代号...
   2\t...
   ```
   特征：列间用 `\t` 分隔，行间用 `\n` 分隔，首行为表头。每行是独立条目，但需要表头定义列含义。

2. **密集清单** — 车驾管服务网点名单（2,231 字符，32 行）：
   ```
   序号业务网点名称详细地址业务类型类型咨询电话
   1芜湖市辉泰机动车检测有限公司安徽省芜湖市鸠江区...
   2芜湖市广安机动车检测服务有限公司芜湖经济技术开发区...
   ```
   特征：无 `\t`，但每行是一个独立条目（一个网点），首行为表头。行长度均匀（60-90 字符），与普通散文的长短不一形成对比。

3. **普通文本** — 法规前言（5,096 字符）：
   ```
   本标准的全部技术内容为强制性，按照 GB/T 1.1—2009 给出的规则起草...
   ```
   特征：连续散文，含 `。`、`；`、`\n` 等自然语言分隔符，无表格结构。

**为什么密集清单需要独立管线？** 若走普通文本管线，分隔符 `。`、`；` 会在条目内部切分（如网点名称、地址中可能含句号），把一个完整条目切断。走表格管线按 `\n` 分组则每个 chunk 包含若干完整条目。

**制表符表格切分：** 按 `\n` 拆分为行，累积数据行直到加入下一行会超过 `CHUNK_SIZE`，然后开始新 chunk。每个子 chunk 头部附加原始表头行。跳过续表标记行（如"表A.1（续）"及其后的重复表头行）。无 overlap。

**密集清单切分：** 按 `\n` 拆分为行，过滤空行和 `---` 分隔线，累积条目行直到接近 `CHUNK_SIZE`。每个子 chunk 头部附加表头行（首行）。无 overlap。

**普通文本切分：优先级分隔符递归切分**

分隔符优先级（从粗到细）：`["\n\n", "。", "；", "\n", "，", " "]`

> **为什么 `。` 优先于 `\n`？** 法规文本中存在"引导句 + 条款列表"模式（如"办理变更备案的业务流程为：\n（一）...\n（二）..."），此时 `\n` 是条款边界而非语义断裂点。按 `\n` 切分会把引导句单独切断，失去上下文。先按 `。` 切可保持完整语义块。

流程：
1. 取当前最高优先级的分隔符，将文本切分为片段
2. 贪心合并小片段到当前 chunk，直到加入下一片段会超过 `CHUNK_SIZE`
3. 仍超限的片段用下一优先级分隔符递归切分
4. 递归直到所有片段都 ≤ `CHUNK_SIZE`，或已用尽所有分隔符（最后兜底按字符截断）

**Overlap：** 仅当实际使用的分隔符为 `" "`（空格）或兜底字符截断时，才在相邻 chunk 之间添加 `CHUNK_OVERLAP` 重叠。高优先级分隔符（`\n\n`、`。`、`；`、`\n`）本身就是自然语义边界，不需要 overlap。制表符表格和密集清单均不使用 overlap（按行分组+表头保留，不存在生硬截断）。

**参数（待确认）：**
- `CHUNK_SIZE`：触发切分的长度阈值，建议 512 字符
- `CHUNK_OVERLAP`：相邻 chunk 重叠字符数，仅在低语义强度分隔符时使用，建议 50-100

**持久化策略：** 切分结果持久化到 `artifacts/temp/split_metadatas.json`，不修改原始 `metadatas.json`。这样：
- 原始数据不动，无需同步数据团队
- 切分结果可见可调试
- 建索引时读切分后的 JSON，而非原始数据

> **元数据继承：** 切分产生的每个子 chunk 作为独立文档进入索引，继承原文档的元数据（文件名、章节、条款号），新增 chunk 序号字段以标识来源。
